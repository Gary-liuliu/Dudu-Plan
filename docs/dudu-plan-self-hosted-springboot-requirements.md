# 嘟嘟计划自建实时服务开发需求说明

## 1. 项目现状

- 原网站后端使用 Spring Boot、Maven、Java 17。
- 服务监听 `127.0.0.1:9191`，部署文件为：

  ```text
  /xyjh/XYJHStart-Web-0.0.1-SNAPSHOT.jar
  ```

- 网站域名为 `https://kakaweb.ltd`。
- Nginx 已将 `/api/` 转发到 `127.0.0.1:9191`，并包含 WebSocket Upgrade 配置。
- 原项目当前没有登录校验和 WebSocket 功能。
- 嘟嘟计划当前使用 Cloudflare Worker 做实时中继，使用 Firebase/FCM 发送远程通知。

## 2. 改造目标

在原 Spring Boot 项目中增加独立的“嘟嘟计划”模块，替代 Cloudflare Worker 和 Firebase。

- 不安装 Docker，不新增 Java 进程。
- 不改变原网站已有接口，不给原网站增加全局登录校验。
- 只增加一张聊天消息表，不使用 Redis，不把训练数据写入数据库。
- 服务端在内存中保存当前 WebSocket 连接，在数据库中保存聊天消息和送达状态。
- 训练历史继续保存在嘟嘟手机，肚肚手机继续保存已同步缓存。
- `嘟嘟`固定对应 `owner`，可训练和修改记录。
- `肚肚`固定对应 `observer`，训练数据只读，但可以聊天和发送鼓励。
- 采用“App 打开时实时通知”方案；肚肚 App 关闭后不保证收到开始或结束提醒。
- 两个账号支持一对一纯文字聊天，并支持离线消息和历史记录。

## 3. 接口要求

App 基础地址：

```text
https://kakaweb.ltd/api/dudu-plan
```

新增接口：

```text
POST /api/dudu-plan/auth/login
POST /api/dudu-plan/auth/refresh
GET  /api/dudu-plan/health
GET  /api/dudu-plan/chat/messages
GET  /api/dudu-plan/chat/unread-count
POST /api/dudu-plan/chat/delivered
POST /api/dudu-plan/chat/read
POST /api/dudu-plan/chat/messages/{messageId}/recall
WS   /api/dudu-plan/realtime
```

登录请求：

```json
{
  "username": "嘟嘟",
  "password": "用户输入的密码"
}
```

登录和刷新成功响应需要与现有 App 保持兼容：

```json
{
  "accountName": "嘟嘟",
  "role": "owner",
  "accessToken": "...",
  "accessTokenExpiresAt": 0,
  "refreshToken": "...",
  "refreshTokenExpiresAt": 0
}
```

- 时间使用 Unix 毫秒。
- Access Token 有效期为 15 分钟。
- Refresh Token 有效期为 365 天。当前版本采用无状态签名令牌，刷新时签发新 Token，但旧 Refresh Token 在到期前仍然有效；需要立即使全部令牌失效时轮换服务端签名密钥。
- 错误账号和错误密码统一返回 `401` 与 `{"error":"invalid_credentials"}`。
- 登录接口需要进行内存限速，连续失败时返回 `429`。
- 除登录、刷新和健康检查外，HTTP 接口都使用 `Authorization: Bearer <accessToken>`。
- 聊天历史使用主键游标分页；送达、已读和撤回接口均为幂等操作。
- 常用错误统一为：参数错误 `400`、未登录或 Token 失效 `401`、角色越权 `403`、消息不存在 `404`、幂等键冲突或状态冲突 `409`。

最小接口约定：

- 刷新接口接收 `refreshToken`，成功时返回与登录相同的结构。
- 历史接口接收可选的 `beforeId` 和 `limit`，返回 `items` 与 `nextCursor`；`limit` 默认且最大为 50。
- 送达接口接收 `messageIds`，只允许接收者确认已经保存到本机的消息。
- 已读接口接收 `upToMessageId`，将该接收者此前尚未撤回的消息整体标记为已读。
- 撤回接口成功后返回撤回占位记录；重复撤回返回同一结果。

## 4. 账号与安全要求

- 明文密码不得进入源码、Git、JAR、配置文件或日志。
- 两个固定账号的密码只保存带独立盐值的安全散列，散列和令牌签名密钥通过服务器环境变量注入。
- Access Token 和 Refresh Token 必须区分用途，使用 Refresh Token 不得连接 WebSocket。
- 日志不得记录密码、Token、训练完整消息或请求正文。
- 聊天接口必须校验 Access Token，发送角色从 Token 中获取，不能信任客户端提交的角色字段。
- 新服务签发的令牌与 Cloudflare 旧令牌不兼容，新版 App 需要强制重新登录一次，但不能清除训练数据和观察端缓存。

## 5. WebSocket 要求

连接地址：

```text
wss://kakaweb.ltd/api/dudu-plan/realtime
```

Access Token 不放在 URL 查询参数中。连接建立后，客户端必须在 5 秒内发送以下首帧认证消息：

```json
{
  "protocolVersion": 2,
  "type": "authenticate",
  "accessToken": "..."
}
```

认证成功后服务端返回包含 `role`、`serverTime` 和 `accessTokenExpiresAt` 的 `authenticated`，角色只能从 Access Token 推导。认证超时或失败时关闭连接，分别使用关闭码 `4002` 和 `4003`。Access Token 到期后服务端使用 `4003` 关闭连接，App 刷新 Token 后自动重连；本地训练和记录不得因此中断。

实时协议版本升级为 `2`，消息至少包含：

```json
{
  "protocolVersion": 2,
  "type": "heartbeat",
  "sentAt": 0
}
```

HTTP 与 WebSocket 对外时间字段统一使用 Unix 毫秒。

双方都可发送：

- `chat_message`：发送聊天消息。
- `heartbeat`：连接心跳。

嘟嘟额外可发送：

- `snapshot`：完整训练快照，可由多条分片组成。
- `session_upsert`：单条训练记录更新。
- `workout_event`：开始或结束训练事件。

肚肚额外可发送：

- `event_ack`：确认已收到训练事件。

服务端发送：

- `authenticated`
- `presence`
- `snapshot`
- `session_upsert`
- `workout_event`
- `event_ack`
- `chat_message`
- `chat_saved`
- `chat_delivered`
- `chat_read`
- `chat_recalled`
- `heartbeat`
- `error`

送达、已读和撤回统一通过 HTTP 接口提交，HTTP 与断线重试使用同一幂等规则，不再提供重复的 WebSocket 提交入口。实时事件去向固定为：`chat_saved` 返回消息发送者，`chat_message` 发给接收者，`chat_delivered` 和 `chat_read` 发给原消息发送者，`chat_recalled` 发给另一方，`event_ack` 发给嘟嘟；认证、心跳和错误只返回当前连接。

每次完整快照使用同一个 `snapshotId`，并在每条 `snapshot` 消息中包含 `chunkIndex`、`chunkCount` 和该分片的 `sessions`。单条分片不得超过 64 KiB；接收方收齐全部分片后，原子地按 `sessionId + updatedAt` 合并到缓存，不得整体覆盖较新的本地记录。重复分片按 `snapshotId + chunkIndex` 去重，缺失或超时的快照整体丢弃，等待下一次重连补齐。

连接规则：

- 每个角色最多保留一个连接，同角色新连接建立后以关闭码 `4001` 替换旧连接。
- 训练数据和聊天正文只转发给另一角色；确认和状态事件严格按照上方事件去向发送，不做无条件广播。
- 训练快照和训练增量不持久化；聊天消息必须先写入数据库再确认发送成功。
- 无论哪一方后建立连接，只要双方均在线，服务端都通知嘟嘟发送一次完整快照；肚肚从离线变为在线时再次触发快照。
- `WorkoutSession` 必须包含 `updatedAt`；切换动作、修改组数据、开始休息和结束训练时都更新该字段。
- 心跳间隔约 25 秒，超时连接由服务端关闭。
- 只接受文本 JSON，单条消息最大 64 KiB。
- 服务端仅验证协议、角色权限和基本结构，不解析或修改训练业务数据。
- 替换同角色旧连接时，旧连接关闭回调只有在其仍为当前连接时才能移除角色映射，避免误删新连接。

## 6. 通知要求

- 新版 App 和新后端不再调用 Firebase、FCM、Expo Push Token 或远程 `/push` 接口。
- 嘟嘟开始或完成训练时，通过 WebSocket 发送 `workout_event`。
- 肚肚 App 打开且在线时显示页面提示或安卓本地通知。
- 事件 ID 使用 `sessionId + eventType`，肚肚手机本地去重。
- 嘟嘟手机可在本地保留未确认事件最多 30 分钟，收到 `event_ack` 后标记完成。
- 不要求肚肚 App 被关闭、强制停止或清理后台后继续接收提醒。

## 7. 聊天功能要求

### 7.1 功能范围

- 仅支持嘟嘟和肚肚之间的一对一聊天，不支持群聊。
- 第一版支持纯文字和 Emoji，去除首尾空白后不得为空，单条消息最多 2000 个 Unicode 字符且整个实时消息仍受 64 KiB 限制，不解析 HTML。
- 支持离线消息、历史记录分页、未读数量、已送达、已读、回复引用和两分钟内撤回。
- 暂不支持图片、语音、视频、文件、消息编辑和同角色多设备同时在线；新设备登录后仍可从服务器拉取已有聊天历史。
- 聊天页面显示发送中、已保存、已送达、已读和发送失败状态。
- App 打开时可显示未读角标、站内提示或本地通知；App 关闭后不保证实时提醒，但消息仍保存在数据库中。

### 7.2 数据表

只增加一张 `dudu_chat_message` 表，数据库字符集使用 `utf8mb4`。主要字段：

```text
id                    数据库自增主键
message_id            客户端生成的唯一 UUID，建立唯一索引
sender_role           owner 或 observer
receiver_role         owner 或 observer
message_type          text 或 encouragement
content               消息正文
reply_to_message_id   被回复消息的 message_id，可为空
client_created_at     客户端创建时间
server_created_at     服务端接收时间
delivered_at          送达设备时间，可为空
read_at               阅读时间，可为空
recalled_at           撤回时间，可为空
```

需要为 `message_id` 建立唯一索引，并为接收角色、主键和未读查询建立组合索引。训练记录、饮食、饮水、身体资料和 App 设置不得写入该表。

### 7.3 消息流程

- 发送方先把消息保存到本机，再通过 `chat_message` 提交。
- 服务端从 Token 获取发送角色，自动确定接收角色，并按 `message_id` 幂等写入数据库。
- 数据库保存成功后，服务端向发送方返回 `chat_saved`；接收方在线时同时转发消息。
- 接收方保存成功后调用送达接口，打开聊天页面并看到消息后调用已读接口；服务端将状态变化实时通知发送方。
- 对方离线时消息继续保存在数据库中，对方下次打开 App 后通过历史接口获取。
- 客户端断线重发同一个 `message_id` 时不得生成重复记录。重复 ID 仅在发送角色、接收角色、类型、正文和 `reply_to_message_id` 全部一致时返回原记录，否则返回 `409`。
- 只有接收者可以标记送达和已读，只有发送者可以撤回自己的消息；状态时间只能前进不能回退，已读自动包含送达。
- 发送方可在服务端接收消息后的两分钟内撤回自己的消息；撤回后记录 `recalled_at`，客户端显示“撤回了一条消息”。
- 历史接口和实时事件对已撤回消息只返回撤回占位，不再返回原正文；已撤回消息不计入未读数量。
- 历史接口默认每页 50 条，使用主键游标向前分页，不使用页码深分页。
- 聊天排序以数据库主键和 `server_created_at` 为准，手机时间仅用于展示客户端发送时间。
- 服务端和 Nginx 日志不得记录聊天正文。

### 7.4 心形鼓励功能

- 肚肚首页的“正在训练”卡片增加心形鼓励按钮，仅在嘟嘟在线且存在未结束训练时启用；离线时可通过普通聊天留言。
- 点击心形按钮后弹出简洁的预设文案列表，选择后作为 `message_type=encouragement` 的聊天消息发送并写入同一张聊天表。
- 鼓励消息复用聊天的离线保存、送达、已读和去重逻辑，不新增数据表。
- 嘟嘟在线训练时收到浮动爱心和短文字提示，提示自动消失，不遮挡动作记录和休息倒计时。
- 实时状态变化后若发送失败，鼓励消息仍按普通聊天消息重试并保存在数据库中，下次打开 App 后可在聊天记录查看。
- 心形按钮采用 3 秒点击冷却，服务端同时按账号限速，避免连续点击或修改版客户端产生大量重复消息。
- 自定义文案通过普通聊天发送，鼓励按钮第一版只提供预设内容。

## 8. App 改造要求

- 实时服务地址改为 `https://kakaweb.ltd/api/dudu-plan`。
- 新版 App 停止调用 Cloudflare Worker、Firebase、FCM 和 Expo Push，并且不把对应凭据打入新 APK；旧云端资源暂时保留供旧 APK 回滚。
- 保留训练提醒和前台本地通知能力。
- WebSocket 改为建连后发送认证消息，不再把 Access Token 放在 URL 中。
- 账号安全存储键升级，安装新版后要求重新登录一次。
- 原有训练数据、营养数据和肚肚本地缓存不得删除。
- 两个账号都增加聊天入口、未读角标、消息气泡、时间分组、下拉加载历史和长按回复/复制/撤回。
- 手机本地缓存最近聊天记录，登录后从服务器同步缺失消息。
- 肚肚不增加底部导航，在首页顶部提供聊天入口和未读角标；训练详情仍保持只读。
- 肚肚的实时训练卡片增加心形鼓励按钮；嘟嘟训练页增加鼓励消息浮层和爱心动画。
- App 版本升级为 `1.2.0`，Android `versionCode` 升为 `4`。

### 8.1 升级与本地数据迁移

- 新版只强制两个账号重新登录，不清除训练、饮水、蛋白质、设置或肚肚观察端缓存。
- 保留旧版已经建立的训练同步起始边界；边界之前的历史继续只保存在嘟嘟手机，不因切换服务器而补传。
- 升级时正在进行的训练允许作为实时快照同步。
- 同一训练会话按 `sessionId` 合并，冲突时以更大的 `updatedAt` 为准，不允许旧快照覆盖较新的本地记录。
- 模板 v1 和 v2 的历史会话继续按各自保存的模板版本渲染；缺少版本的旧会话按 v1 处理。
- 嘟嘟与肚肚的本地缓存必须按角色隔离，退出或切换账号不能串用数据。

## 9. 训练数据 Markdown 导出

### 9.1 功能目标

- 仅在嘟嘟账号提供“导出训练数据”入口，放在训练记录页面或设置页面中。
- 所有导出计算和 Markdown 生成均在手机本地完成，不上传服务器。
- 支持导出最近 4 周、8 周、12 周、全部记录或自定义日期范围，默认选择最近 4 周。
- 支持预览、复制 Markdown 文本和分享 `.md` 文件。
- 文件使用 UTF-8 编码，文件名格式为：

  ```text
  dudu-training-开始日期_结束日期.md
  ```

- 默认只导出已完成训练。周计划次数按日期范围内实际包含的周一、周二、周四和周五计算，部分周只计算落在范围内的计划日，不推测跳过或补练。
- 同时导出日期范围内的饮水量和蛋白质摄入量，以及导出时的当前饮水和蛋白质目标；历史达标情况统一按该目标回算并在文档中注明。
- 不导出账号、Token、聊天、身体围度或其他 App 设置。

### 9.2 导出内容

Markdown 需要包含：

- 导出时间、日期范围、时区和包含的训练模板版本。
- 每周计划频率、实际完成次数、总训练时长、完成组数和不适组数。
- 每周胸、背、肩、腿和核心等分组的已完成有效组数。
- 动作训练次数、完成组数、最近使用重量、次数或支撑秒数。
- 每次训练的日期、训练类型、开始/结束时间、训练时长和完成状态。
- 每个动作的组号、重量、次数或秒数、RIR、完成状态和不适记录。
- 每日饮水总量、蛋白质总量、目标值、达标情况和当天是否训练。
- 日期范围内的饮水与蛋白质记录天数、日均摄入和达标天数。
- 明确说明哑铃重量按单只哑铃记录，标记“每侧”的动作按单侧次数理解。
- 不自动计算总吨位，避免双哑铃、单哑铃和每侧动作产生错误倍数。

训练时长使用 `completedAt - startedAt` 计算，跨日训练归入 `startedAt` 所在的本地日期；旧训练必须按其保存的模板版本恢复动作名称，缺少模板版本的旧会话按 v1 处理。每次训练明细都标明实际模板版本。

重量、次数、秒数、RIR 和不适记录没有值时必须输出“未记录”，不能用 `0` 或“否”代替。有效组数仅统计标记为完成的组，不根据动作名称自动推测未记录的组。

饮水和蛋白质只统计用户实际记录的条目。某天没有任何条目时显示“未记录”，不能当作摄入量为 0，也不能计入日均值或未达标天数。

### 9.4 导出与测试要求

- 无训练记录时也能生成包含说明和空状态的合法 Markdown。
- 导出顺序固定为日期从旧到新，动作与组按训练时顺序排列。
- 中文、Emoji、换行和 Markdown 特殊字符必须正确转义或安全输出。
- 大量历史优先通过文件分享，不要求全部复制到剪贴板。
- 生成文件使用 App 缓存目录和安卓系统分享面板，不申请不必要的存储权限。
- 单元测试覆盖日期范围、周汇总、时长计算、v1/v2 模板、重量/次数/秒数、RIR、不适记录、饮水/蛋白质汇总、未记录日期和敏感字段排除。

## 10. Nginx 与部署要求

- 继续使用现有 `/api/` 反向代理和 `9191` 端口，无需新增公网端口。
- `9191` 只允许本机访问，腾讯云安全组不开放该端口。
- 现有 WebSocket Upgrade 配置可以复用，实时接口建议关闭代理缓冲并保留长连接超时。
- 建议将 TLS 配置调整为仅允许 TLS 1.2 和 TLS 1.3。
- 发布前备份原 JAR 和 Nginx 配置，再替换：

  ```text
  /xyjh/XYJHStart-Web-0.0.1-SNAPSHOT.jar
  ```

- 发布顺序固定为：先部署兼容旧网站的新后端，再发布新 APK；两台手机验证通过后，才停用并删除 Cloudflare、Firebase 资源和相关私钥。
- 发布新 JAR 前先执行聊天表迁移，并备份现有数据库；表结构变更必须兼容旧网站。

## 11. 验收标准

- 原网站页面、接口和 `9191` 部署方式不受影响，新功能仅位于 `/api/dudu-plan`。
- 登录、刷新、权限校验、限速和日志脱敏符合本文约束，升级登录不会清除任何本地业务数据。
- 双方在线时训练增量实时可见；断线、Token 到期或服务重启后能够自动重连，并通过完整分片快照恢复一致状态。
- 模板 v1/v2、同步起始边界和 `updatedAt` 合并规则通过升级测试，旧历史不会被补传或覆盖。
- 聊天支持在线发送、离线历史、游标分页、未读、送达、已读、回复和两分钟撤回，重复请求不会产生重复记录或状态回退。
- 肚肚只能读取训练数据，但可以聊天并在嘟嘟在线训练时发送受限速保护的预设鼓励。
- App 关闭后不承诺实时提醒；重新打开后聊天历史和训练缓存仍可恢复。
- Markdown 导出在相同数据和范围下结构、排序及统计结果稳定，缺失值、模板版本、训练分组、饮水和蛋白质口径表达明确。
- 服务端数据库只保存聊天记录；训练、营养、身体资料和 App 设置不会上传。
- 核心测试覆盖快照分片、Token 到期重连、账号切换缓存隔离、数据库失败重试、已读与撤回竞争，以及 `360×800`、`412×915` 安卓视口。
