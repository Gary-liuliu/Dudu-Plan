# 嘟嘟计划自建实时服务开发需求说明

## 1. 项目现状

- 原网站后端使用 Spring Boot、Maven、Java 17。
- 服务监听 `127.0.0.1:9191`，部署文件为：

  ```text
  /xyjh/XYJHStart-Web-0.0.1-SNAPSHOT.jar
  ```

- 网站域名为 `https://kakaweb.ltd`。
- Nginx 已将 `/api/` 转发到 `127.0.0.1:9191`，并包含 WebSocket Upgrade 配置。
- 原项目当前没有登录校验和 WebSocket 功能。
- 嘟嘟计划当前使用 Cloudflare Worker 做实时中继，使用 Firebase/FCM 发送远程通知。

## 2. 改造目标

在原 Spring Boot 项目中增加独立的“嘟嘟计划”模块，替代 Cloudflare Worker 和 Firebase。

- 不安装 Docker，不新增 Java 进程。
- 不改变原网站已有接口，不给原网站增加全局登录校验。
- 只增加一张聊天消息表，不使用 Redis，不把训练数据写入数据库。
- 服务端在内存中保存当前 WebSocket 连接，在数据库中保存聊天消息和送达状态。
- 训练历史继续保存在嘟嘟手机，肚肚手机继续保存已同步缓存。
- `嘟嘟`固定对应 `owner`，可训练和修改记录。
- `肚肚`固定对应 `observer`，只能查看记录。
- 采用“App 打开时实时通知”方案；肚肚 App 关闭后不保证收到开始或结束提醒。
- 两个账号支持一对一纯文字聊天，并支持离线消息和历史记录。

## 3. 接口要求

App 基础地址：

```text
https://kakaweb.ltd/api/dudu-plan
```

新增接口：

```text
POST /api/dudu-plan/auth/login
POST /api/dudu-plan/auth/refresh
GET  /api/dudu-plan/health
GET  /api/dudu-plan/chat/messages
GET  /api/dudu-plan/chat/unread-count
POST /api/dudu-plan/chat/read
POST /api/dudu-plan/chat/messages/{messageId}/recall
WS   /api/dudu-plan/realtime
```

登录请求：

```json
{
  "username": "嘟嘟",
  "password": "用户输入的密码"
}
```

登录和刷新成功响应需要与现有 App 保持兼容：

```json
{
  "accountName": "嘟嘟",
  "role": "owner",
  "accessToken": "...",
  "accessTokenExpiresAt": 0,
  "refreshToken": "...",
  "refreshTokenExpiresAt": 0
}
```

- 时间使用 Unix 毫秒。
- Access Token 有效期建议为 15 分钟。
- Refresh Token 有效期建议为 365 天并在刷新时轮换。
- 错误账号和错误密码统一返回 `401` 与 `{"error":"invalid_credentials"}`。
- 登录接口需要进行内存限速，连续失败时返回 `429`。

## 4. 账号与安全要求

- 明文密码不得进入源码、Git、JAR、配置文件或日志。
- 两个固定账号的密码只保存带独立盐值的安全散列，散列和令牌签名密钥通过服务器环境变量注入。
- Access Token 和 Refresh Token 必须区分用途，使用 Refresh Token 不得连接 WebSocket。
- 日志不得记录密码、Token、训练完整消息或请求正文。
- 聊天接口必须校验 Access Token，发送角色从 Token 中获取，不能信任客户端提交的角色字段。
- 新服务签发的令牌与 Cloudflare 旧令牌不兼容，新版 App 需要强制重新登录一次，但不能清除训练数据和观察端缓存。

## 5. WebSocket 要求

连接地址：

```text
wss://kakaweb.ltd/api/dudu-plan/realtime
```

Access Token 不放在 URL 查询参数中。连接建立后，客户端必须在 5 秒内发送认证消息，认证成功后才允许注册角色和转发数据。

实时协议版本升级为 `2`，消息至少包含：

```json
{
  "protocolVersion": 2,
  "type": "heartbeat",
  "sentAt": "ISO-8601时间"
}
```

嘟嘟可发送：

- `snapshot`：完整训练快照。
- `session_upsert`：单条训练记录更新。
- `workout_event`：开始或结束训练事件。
- `chat_message`：发送聊天消息。
- `chat_delivered`：确认消息已送达设备。
- `chat_read`：提交已读状态。
- `chat_recall`：申请撤回消息。
- `heartbeat`：连接心跳。

肚肚可发送：

- `event_ack`：确认已收到训练事件。
- `chat_message`：发送聊天消息。
- `chat_delivered`：确认消息已送达设备。
- `chat_read`：提交已读状态。
- `chat_recall`：申请撤回消息。
- `heartbeat`：连接心跳。

服务端发送：

- `authenticated`
- `presence`
- `snapshot`
- `session_upsert`
- `workout_event`
- `chat_message`
- `chat_saved`
- `chat_delivered`
- `chat_read`
- `chat_recalled`
- `heartbeat`
- `error`

连接规则：

- 每个角色最多保留一个连接，同角色新连接建立后以关闭码 `4001` 替换旧连接。
- 消息只转发给另一角色，不回发给发送者。
- 训练快照和训练增量不持久化；聊天消息必须先写入数据库再确认发送成功。
- 肚肚重新上线后，嘟嘟发送完整快照补齐缓存。
- 心跳间隔约 25 秒，超时连接由服务端关闭。
- 只接受文本 JSON，单条消息最大 64 KiB。
- 服务端仅验证协议、角色权限和基本结构，不解析或修改训练业务数据。

## 6. 通知要求

- 移除 Firebase、FCM、Expo Push Token 和远程 `/push` 接口。
- 嘟嘟开始或完成训练时，通过 WebSocket 发送 `workout_event`。
- 肚肚 App 打开且在线时显示页面提示或安卓本地通知。
- 事件 ID 使用 `sessionId + eventType`，肚肚手机本地去重。
- 嘟嘟手机可在本地保留未确认事件最多 30 分钟，收到 `event_ack` 后标记完成。
- 不要求肚肚 App 被关闭、强制停止或清理后台后继续接收提醒。

## 7. 聊天功能要求

### 7.1 功能范围

- 仅支持嘟嘟和肚肚之间的一对一聊天，不支持群聊。
- 第一版支持纯文字和 Emoji，单条消息最多 2000 个字符，不解析 HTML。
- 支持离线消息、历史记录分页、未读数量、已送达、已读、回复引用和两分钟内撤回。
- 暂不支持图片、语音、视频、文件、消息编辑和多设备历史合并。
- 聊天页面显示发送中、已保存、已送达、已读和发送失败状态。
- App 打开时可显示未读角标、站内提示或本地通知；App 关闭后不保证实时提醒，但消息仍保存在数据库中。

### 7.2 数据表

只增加一张 `dudu_chat_message` 表，数据库字符集使用 `utf8mb4`。主要字段：

```text
id                    数据库自增主键
message_id            客户端生成的唯一 UUID，建立唯一索引
sender_role           owner 或 observer
receiver_role         owner 或 observer
message_type          当前固定为 text
content               消息正文
reply_to_message_id   被回复消息的 message_id，可为空
client_created_at     客户端创建时间
server_created_at     服务端接收时间
delivered_at          送达设备时间，可为空
read_at               阅读时间，可为空
recalled_at           撤回时间，可为空
```

需要为 `message_id` 建立唯一索引，并为接收角色、主键和未读查询建立组合索引。训练记录、饮食、饮水、身体资料和 App 设置不得写入该表。

### 7.3 消息流程

- 发送方先把消息保存到本机，再通过 `chat_message` 提交。
- 服务端从 Token 获取发送角色，自动确定接收角色，并按 `message_id` 幂等写入数据库。
- 数据库保存成功后，服务端向发送方返回 `chat_saved`；接收方在线时同时转发消息。
- 接收方保存成功后发送 `chat_delivered`，打开聊天页面并看到消息后发送 `chat_read`。
- 对方离线时消息继续保存在数据库中，对方下次打开 App 后通过历史接口获取。
- 客户端断线重发同一个 `message_id` 时不得生成重复记录。
- 发送方可在服务端接收消息后的两分钟内撤回自己的消息；撤回后记录 `recalled_at`，客户端显示“撤回了一条消息”。
- 历史接口默认每页 50 条，使用主键游标向前分页，不使用页码深分页。
- 服务端和 Nginx 日志不得记录聊天正文。

## 8. App 改造要求

- 实时服务地址改为 `https://kakaweb.ltd/api/dudu-plan`。
- 删除 Cloudflare Worker、Firebase、FCM 和 Expo Push 相关代码及配置。
- 保留训练提醒和前台本地通知能力。
- WebSocket 改为建连后发送认证消息，不再把 Access Token 放在 URL 中。
- 账号安全存储键升级，安装新版后要求重新登录一次。
- 原有训练数据、营养数据和肚肚本地缓存不得删除。
- 两个账号都增加聊天入口、未读角标、消息气泡、时间分组、下拉加载历史和长按回复/复制/撤回。
- 手机本地缓存最近聊天记录，登录后从服务器同步缺失消息。
- App 版本升级为 `1.2.0`，Android `versionCode` 升为 `4`。

## 9. 训练数据 Markdown 导出

### 9.1 功能目标

- 仅在嘟嘟账号提供“导出训练数据”入口，放在训练记录页面或设置页面中。
- 所有导出计算和 Markdown 生成均在手机本地完成，不上传服务器。
- 支持导出最近 4 周、8 周、12 周、全部记录或自定义日期范围，默认选择最近 4 周。
- 支持预览、复制 Markdown 文本和分享 `.md` 文件。
- 文件使用 UTF-8 编码，文件名格式为：

  ```text
  dudu-training-开始日期_结束日期.md
  ```

- 默认导出已完成训练，同时在周汇总中体现计划次数、跳过次数和补练次数。
- 不导出账号、Token、聊天、饮食、饮水、身体围度或 App 设置。

### 9.2 导出内容

Markdown 需要包含：

- 导出时间、日期范围、时区和训练模板版本。
- 每周计划频率、实际完成次数、总训练时长、完成组数和不适组数。
- 动作训练次数、完成组数、最近使用重量、次数或支撑秒数。
- 每次训练的日期、训练类型、开始/结束时间、训练时长和完成状态。
- 每个动作的组号、重量、次数或秒数、RIR、完成状态和不适记录。
- 明确说明哑铃重量按单只哑铃记录，标记“每侧”的动作按单侧次数理解。
- 不自动计算总吨位，避免双哑铃、单哑铃和每侧动作产生错误倍数。

训练时长使用 `completedAt - startedAt` 计算；旧训练必须按其保存的模板版本恢复动作名称，缺少模板版本的旧会话按 v1 处理。

### 9.3 Markdown 格式

导出格式保持稳定，便于直接发送给 AI：

```markdown
# 嘟嘟计划训练数据

- 导出时间：2026-07-16 20:30 GMT+8
- 数据范围：2026-06-19 至 2026-07-16
- 计划频率：每周 4 练
- 记录约定：哑铃重量为单只重量；“每侧”动作按单侧次数记录

## 周汇总

| 周期 | 计划次数 | 完成次数 | 补练 | 总时长 | 完成组数 | 不适组数 |
|---|---:|---:|---:|---:|---:|---:|
| 2026-07-13 至 2026-07-19 | 4 | 3 | 0 | 02:18:40 | 45 | 0 |

## 动作汇总

| 动作 | 分组 | 训练次数 | 完成组数 | 最近重量 | 最近表现 |
|---|---|---:|---:|---:|---|
| 哑铃高脚杯深蹲 | 力量训练 | 3 | 12 | 20 kg | 12、12、11、10 次 |
| 平板支撑 | 核心训练 | 2 | 6 | 自重 | 60、60、55 秒 |

## 训练明细

### 2026-07-14 · 下肢 A

- 状态：已完成
- 训练时间：19:30-20:18
- 训练时长：00:48:00

#### 哑铃高脚杯深蹲

| 组 | 重量 | 次数/时间 | RIR | 不适 | 状态 |
|---:|---:|---:|---:|---|---|
| 1 | 20 kg | 12 次 | 2 | 否 | 完成 |

## 希望 AI 分析的问题

1. 当前每周训练频率和总组数是否足够支持增肌？
2. 胸、背、肩、腿和核心的训练量是否均衡？
3. 重量、次数和 RIR 是否体现稳定进步？
4. 是否存在恢复不足、动作重复或训练量过高的问题？
5. 下一阶段应优先增加重量、次数还是训练组数？
```

### 9.4 导出与测试要求

- 无训练记录时也能生成包含说明和空状态的合法 Markdown。
- 导出顺序固定为日期从旧到新，动作与组按训练时顺序排列。
- 中文、Emoji、换行和 Markdown 特殊字符必须正确转义或安全输出。
- 大量历史优先通过文件分享，不要求全部复制到剪贴板。
- 生成文件使用 App 缓存目录和安卓系统分享面板，不申请不必要的存储权限。
- 单元测试覆盖日期范围、周汇总、时长计算、v1/v2 模板、重量/次数/秒数、RIR、不适记录和敏感字段排除。

## 10. Nginx 与部署要求

- 继续使用现有 `/api/` 反向代理和 `9191` 端口，无需新增公网端口。
- `9191` 只允许本机访问，腾讯云安全组不开放该端口。
- 现有 WebSocket Upgrade 配置可以复用，实时接口建议关闭代理缓冲并保留长连接超时。
- 建议将 TLS 配置调整为仅允许 TLS 1.2 和 TLS 1.3。
- 发布前备份原 JAR 和 Nginx 配置，再替换：

  ```text
  /xyjh/XYJHStart-Web-0.0.1-SNAPSHOT.jar
  ```

- 新后端和新 APK 验证完成前，保留现有 Cloudflare 和 Firebase 配置作为回滚方案。
- 发布新 JAR 前先执行聊天表迁移，并备份现有数据库；表结构变更必须兼容旧网站。

## 11. 验收标准

- 原网站现有页面和接口不受影响。
- 正确账号可登录，错误账号或密码返回统一错误。
- 嘟嘟与肚肚同时打开 App 时能建立实时连接并显示在线状态。
- 开始训练、切换动作、修改组数据、休息计时和结束训练可以实时同步。
- 断网重连后由完整快照补齐，不产生重复训练记录。
- 同角色重复连接会替换旧连接。
- 非法消息、越权消息和超过 64 KiB 的消息会被拒绝。
- 服务端重启后不保留训练数据，手机本地训练功能仍可正常使用。
- 肚肚 App 打开时能收到开始和结束训练提示；App 关闭时不做通知保证。
- 双方同时在线时可以实时发送文字消息并收到送达确认。
- 对方离线时消息保存到数据库，对方下次打开 App 后可以获取。
- 聊天历史支持分页、未读数、已送达、已读、回复引用和两分钟撤回。
- 客户端重复提交同一 `message_id` 时数据库只保留一条消息。
- Java 服务重启后聊天历史不丢失，训练数据仍不在服务器保存。
- 嘟嘟可按日期范围生成 Markdown，内容准确包含训练时长、动作和每组重量等记录。
- 导出的 Markdown 可预览、复制或分享，且不包含账号、Token、聊天和饮食数据。
- 同一批数据重复导出时结构和排序保持稳定，能够直接交给 AI 分析。
- 日志中不得出现明文密码、Access Token、Refresh Token 或训练完整数据。
